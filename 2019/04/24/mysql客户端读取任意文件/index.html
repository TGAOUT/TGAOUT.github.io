利用MySQL LOAD DATA读取任意文件

load data infile

mysql中的load data infile 主要有两种用法。

它们的区别在于

load data infile 读取服务端的文件 （被动连接数据库的一端为服务端）并存入指定表中。

load data local infile 读取客户端的文件 （主动连接数据库的一端为客户端）并存入指定表中。

官方手册上已经给出了关于load data local安全的详细说明

数据包分析

前奏

  1. tcpdump 抓包

    tcpdump -i lo port 3306 -w test.cap

  2. 登录mysql 执行load data local infile ‘filename’ into table tablename

将数据包test.cap拖到本地。利用wireshark进行分析。

分析

  1. 服务端向客户端发送greeting问候数据包，包含banner信息。

  2. 客户端向服务端发送登录请求包，包含用户名和密码，以及和关于load data local 客户端的banner信息。

  3. 然后是客户端初始化的一些查询，比如select database(); select @@version_comment;

  4. 找到我们执行的LOAD DATA INFILE数据包，是客户端发起的Request Query。

  5. 紧接着服务器返回了一个包含刚才所要LOAD DATA INFILE的文件名/etc/passwd的数据包 。

  6. 然后客户端向服务端发送了/etc/passwd文件的内容

结论

如果我们在客户端发送查询之后，返回一个Response TABULAR数据包，即服务端向客户端发送了文件名的数据包，如果我们把这个文件名设置成我们想要读取的文件，那么我们就可以读取客户端的任意文件了。正如官方文档所写的In theory, a patched server could be built that would tell the client program to transfer a file of the server's choosing rather than the file named by the client in the [LOAD DATA](https://dev.mysql.com/doc/refman/8.0/en/load-data.html "13.2.7 LOAD DATA Syntax") statement.

也就是说想要读取哪个文件是服务端说了算，跟客户端所request的文件名没有关系。

最重要的是伪造的服务端可以在任何时候回复一个file-transfer 请求，在官方文档上，会发现有这么一句：”A patched server could in fact reply with a file-transfer request to any statement, not just LOAD DATA LOCAL”

伪造的服务端可以在任何时候回复一个 file-transfer 请求，不一定非要是在LOAD DATA LOCAL的时候。这里有一点要说明的是，如果想要利用此特性，客户端必须具有 CLIENT_LOCAL_FILES 属性，这一点也会在下文提到的官方数据包格式文档中有所说明，这也是为什么在前面测试的时候需要添加--enable-local-infile的原因。

搭建服务端

主要分为

  • 向 MySQL Client 发送Server Greeting
  • 等待 Client 端发送一个Query Package
  • 回复一个file transfer请求

我们需要知道如何构造File Transfer和Server Greeting数据包，这些包的格式都可以在 MySQL 的官方文档上找到。

具体可以看本文下方的链接。

这里直接贴出github上的一个项目：https://github.com/allyshka/Rogue-MySql-Server

测试

测试环境：

  1. 客户端：win10
  2. 服务端：centos

具体步骤：

  1. 关闭centos上的mysql服务，注意3306端口占用问题。利用lsof -i:3306查看进程，kill pid杀掉此进程。

  2. 在centos运行github上的脚本

  3. 在win上连接centos主机数据库

  4. 查看mysql.log文件
